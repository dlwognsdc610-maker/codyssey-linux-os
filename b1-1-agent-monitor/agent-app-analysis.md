# B1-1 agent-app — 동작 관찰 (동적 분석)

> 대상: `agent-app-linux-x86` (과제 zip x86 빌드). arm64 미확인.
> 방법: 실제 구동 + 앱 자체 출력·RSS·syscall(strace) 관찰 + 실패 주입 재현. 바이너리 디컴파일은 하지 않는다.
> 관련: [과제 원문](spec.md) · [monitor.sh](monitor.sh) · [수행 내역서](submission-notes.md)

## 요지

- 앱은 메모리를 **~256MB 부근까지 올렸다 전부 반납**하고 다시 올리는 **경계 톱니파(sawtooth) 진동자**다. 단조 증가(누수) 아님 — RSS를 시계열로 관찰해 확인.
- 명세엔 "누수(leak)" 표현이 없다. "누수 탐지"는 교육생 구전 — RAMP UP 구간만 관측한 데서 나온 것으로 보임.
- 명세와 배포 바이너리가 어긋나는 지점(§4) — 그 중 프로세스명 불일치가 "환경마다 안 돈다"의 1차 원인.

## 1. 관찰 방법 (재현 절차)

디컴파일 없이, 앱을 실제로 돌리고 밖에서 관찰한다:

1. **부팅 출력** — 앱이 부팅 시 `[1/5]`~`[5/5]` 게이트 결과를 stdout으로 찍는다. 각 단계에 일부러 잘못된 입력을 주입해(실패 주입) 무엇을 검사하는지 관측한다.
2. **메모리 궤적** — 실제 프로세스 RSS를 5초 간격으로 폴링해 시계열로 기록(`/proc/<pid>/smaps_rollup`·`ps`).
3. **CPU·우선순위** — `/proc/<pid>/stat`(utime+stime) 1초 델타, `ps -o ni`로 nice 관측.
4. **syscall 레벨** — `strace`로 파일·소켓·시그널 호출을 필요 구간만 관찰.
5. **프로세스 구조** — `ps`로 onefile 런타임의 자식 프로세스·`/tmp/_MEI*` 전개 관측.

## 2. 앱 구조 (관찰된 동작)

### 2-1. 부팅 5단계 게이트 (앱 stdout + 실패 주입)

앱은 부팅 시 5단계를 순서대로 검사하고, 실패하면 그 단계에서 종료(뒷 단계는 "Skipped"). 각 단계에 잘못된 입력을 주입해 관측한 검사 내용:

| # | 단계 | 검사 (실패 주입으로 관측) |
|:-:|:-----|:-----|
| 1 | User | root로 실행하면 즉시 거부 |
| 2 | Env | `AGENT_PORT`=15034 정확 일치, UPLOAD/KEY 경로 존재 + 경로 비교 |
| 3 | Files | 키파일 내용이 정확히 `agent_api_key_test` |
| 4 | Port | `0.0.0.0:15034` bind 가능 |
| 5 | Log | 로그 디렉토리 쓰기권한 |

### 2-2. 메모리 톱니파 (RSS 시계열 관찰)

**실측(2026-07-18, 실제 프로세스 RSS를 5초 간격 2분 폴링):**
`168→194→220→218→244→270→295(정점,t=35s)→270→270→244→218→193→167→142→116→90→13.5(바닥,t=85s)→65→116→142→142→167→193→218(재상승,t=120s)` MB.

정점 찍고 완전히 바닥까지 내려간 뒤 다시 올라가는 걸 한 주기 그대로 관찰 — **단조 증가(누수) 아님이 관찰 근거로 확정.** ~25MB 계단으로 오르내리고, ~256MB 목표 상한 부근(실측 정점 ~295MB)에서 반납으로 꺾인다. 우선순위는 `ps -o ni`로 nice=10(자기 하향) 관측.

### 2-3. 앱 자체 로그

앱은 자기 로그(`agent_app.log`)를 별도로 남긴다(관측: ~1MB 단위 회전). → 학생이 쓸 `monitor.log`(10MB·10개)와 **별개 파일·별개 정책** — 혼동 금지.

## 3. 실행 성립 조건 (앱이 뜨려면)

일반 계정(root 아님) · 환경변수 5종 정확 · 키파일 내용 일치 · 15034 빈 포트 · 로그 디렉토리 쓰기권한. 하나라도 어긋나면 부팅 5단계에서 즉사(§2-1 관측).

## 4. 명세 ↔ 배포물 불일치 (관측)

| ID | 심각도 | 내용 |
|:--:|:------:|:-----|
| A | 높음 | 명세는 프로세스 이름 `agent_app.py`로 헬스체크. 실제 프로세스명은 `agent-app-linux-x86`. `pgrep agent_app.py`는 빈 결과 → 헬스체크 거짓 `exit 1`. **"환경마다 안 돈다"의 1차 원인.** monitor.sh는 포트 기반 탐지로 우회(§5). |
| B | 개념 | 메모리는 상한에서 반납되므로 단조 증가 판정은 표본 시점에 따라 뒤집힌다. 시계열 관측 필요(§2-2). |
| C | 중 | 경고 `MEM>10%`인데 앱 상한은 절대량 ~256MB → 총 RAM에 따라 경고 여부 달라짐. `CPU>20%`도 nice로 낮게 읽힘. |
| D | 중 | 이 x86 빌드는 GLIBC 2.14까지만 필요(22.04에서 실행됨). 게시판 `GLIBC_2.38 not found`는 더 새 환경에서 구운 다른 빌드. glibc는 하위호환만 되므로 상위에서 구우면 22.04 불가. |
| E | 낮음 | onefile은 `/tmp/_MEIxxxx`에 풀고 자식 프로세스 생성 → `ps`에 PID 둘. PID 개수 기반 판정은 흔들림(관측). |
| F | 높음 | 명세는 `AGENT_KEY_PATH`를 **키 파일 경로**(`.../t_secret.key`)로 쓰라지만, 실제 앱은 **디렉토리(`.../api_keys`)까지만** 기대한다 — 파일 경로를 넣으면 Boot 2단계에서 즉시 `Key Path Mismatch`로 실패(직접 재현·확인). 실측값(디렉토리)으로만 부팅 성공. |
| G | 높음 | 명세·수행 내역서 모두 키 파일명을 `t_secret.key`로 쓰는데, 실제 앱은 `secret.key`를 기대한다 — `t_secret.key`만 만들면 Boot 3단계서 `Missing File`로 실패(직접 재현·확인). |

## 5. monitor.sh 설계 메모

- **프로세스 탐지**: `agent_app.py` 문자열 대신 포트로 — `ss -tlnp`에서 `:15034` LISTEN 라인의 PID 추출. 보강 `pgrep -f agent-app-linux`(패턴을 실제 배포 바이너리명까지 좁혀서 무관한 명령행 오탐 방지).
- **메모리**: `ps` RSS(공유 페이지 중복 계상)보다 `/proc/<pid>/smaps_rollup`의 `Pss`. `%`는 `Pss/MemTotal`. 권한 부족 시 `VmRSS` 폴백.
- **CPU%**: 순간값은 위상 의존 — 앱의 5초 버스트 때문. `/proc/<pid>/stat`의 utime+stime을 1초 간격 두 시점 샘플해 시스템 전체 jiffies 델타 대비로 계산(ps 누적평균은 이 위상성을 감춘다).
- **로그 회전**: 앱의 `agent_app.log`와 혼동 금지, `monitor.log`만 10MB·10개.
- **내역서 가점**: 함정 A~G를 운영 관찰로 기록 = 과제가 요구하는 "운영 문제 추적" 역량.

실제 구현 = [`monitor.sh`](monitor.sh). 실측 완료(2026-07-18, 일반 계정·무권한 환경, 실제 배포 바이너리 `agent-app-linux-x86` 구동):

- Boot 5단계 전부 `[OK]` — 단 F·G를 고친 뒤에만(고치기 전엔 2단계에서 즉시 실패, 재현 확인).
- monitor.sh 헬스체크: 살아있을 때 `[OK]`(자식 PID 포트 기반 탐지, E 우회 확인) / 죽었을 때 `[FAIL]` + `exit 1` 둘 다 재현.
- CPU/MEM 실측: 8초 간격 재실행에서 MEM 1.2%→1.7%로 증가(앱의 RAMP UP과 합치), DISK_USED는 실제 `df` 값과 일치.
- monitor.log 포맷·연속 append 정상. 방화벽 비활성 WARNING 정상 발화.
- 참고: 앱 자체 로그(`agent_app.log`)는 `AGENT_LOG_DIR`를 바꿔도 `/var/log/agent-app/` 고정 경로로 가는 것으로 관측 — 실제 배포(명세 경로)에선 우연히 일치해 안 드러나지만, 다른 경로로 배포하면 콘솔 로깅으로만 폴백한다(비치명적).

## 판정

명세·배포 불일치(A~G) 전부 실측(구동·재현)으로 확인. 앱 동작 자체는 정상 구조(5단계 게이트·상한 반납·포트 바인딩)지만 F·G는 명세를 그대로 따르면 부팅조차 안 되는 수준 — 수행 내역서는 재현으로 확인한 값(`AGENT_KEY_PATH`=디렉토리, 키 파일명=`secret.key`)으로 진행해야 한다. monitor.sh는 실제 앱 대상으로 검증 완료.
